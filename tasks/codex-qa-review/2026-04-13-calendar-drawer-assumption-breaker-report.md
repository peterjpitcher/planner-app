**Prioritised Risks**
1. The spec promises drawer-driven state/done behavior, but the shared drawer has no state editor. That is a hard mismatch, not an edge case.
2. The calendar pill would make the same surface both click target and drag target, unlike PlanBoard’s dedicated drag handle. Touch UX is the main place this will break down.
3. A new drawer-specific `handleDrawerUpdate` would create a third mutation path beside `handleMoveTask` and `handleCompleteTask`, which is likely to drift on rules like `today_section` handling and close/revert behavior.
4. The spec’s “revert on failure” is risky with this drawer because it saves fields independently on blur/change; stale rollback can clobber later successful edits. PlanBoard avoids that by refetching on failure at [PlanBoard.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/plan/PlanBoard.jsx:397).
5. Overflow popover wiring is easy to miss because it is a duplicate render path, not an automatic inheritance from the visible pill path.
