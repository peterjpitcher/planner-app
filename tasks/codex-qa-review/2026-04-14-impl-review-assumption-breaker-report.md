**Findings**
- `AB-001 High` The schema/API contract is broken for `user_settings`. The migration creates `daily_planning_enabled`, `weekly_planning_enabled`, and `planning_snooze_until`, but the API reads and upserts `daily_plan_start`, `daily_plan_end`, `weekly_plan_start`, and `weekly_plan_end`. That makes settings GET/PATCH fail at runtime, and the hook’s first `getUserSettings()` call can abort planning before any window logic runs. - `AB-002 High` `planningWindow` mis-detects the weekly window. `isInsideWindow()` treats `20:05 -> 20:00` as a generic past-midnight wrap, so Sunday 10:00 London is incorrectly “inside” the weekly window; meanwhile the weekly branch only runs on Sunday or Monday-before-end, so Tuesday-Saturday weekly revisits can never happen. The shipped logic currently yields: Sunday 10:00 => `weekly`, Sunday 20:10 => `weekly`, Monday 19:00 => `weekly`, Tuesday 21:00 => `daily`. [isInsideWindow](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/lib/planningWindow.js:30) [weekly branch](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/lib/planningWindow.js:86)

- `AB-003 High` Every weekly window is treated as the Sunday combined flow. `PlanningModal` sets `isSundayCombined = windowType === 'weekly'`, so a normal weekly revisit still becomes “Step 1 of 2”, and `handleFinish()` will record a `daily` session on the second submit. There is no Sunday-specific gate here. [modal init](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:24) [finish flow](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:120) [Monday weekly path](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/lib/planningWindow.js:89)

- `AB-004 High` The Sunday combined flow does not have a source for daily candidates. The hook fetches exactly one candidate payload for the active window type, so on Sunday it requests only `weekly` candidates. After step 1 the modal switches to `step='daily'`, but step 2 renders `tasks.dueTomorrow` and `tasks.undatedThisWeek`, which are never present in the weekly response. - `AB-005 High` The cron timing is wrong in GMT. `vercel.json` schedules both `18:55 UTC` and `19:55 UTC`, both cron routes now accept any London hour `18..20`, and `claimCronRun()` dedupes only by `(operation, run_date)`. In winter, the `18:55 London` run claims the day and the intended `19:55 London` execution is skipped as already run. - `AB-006 Medium` Opening and using the planning modal can trigger Office365 sync. The modal’s section-count and sort-order helpers call `apiClient.getTasks()`, and `/api/tasks` GET always kicks off `maybeAutoSyncOffice365()` in the background.
