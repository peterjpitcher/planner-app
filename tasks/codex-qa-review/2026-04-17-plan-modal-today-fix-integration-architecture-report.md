Medium, pre-existing but surfaced by Claude’s counter: manual planning can unmount the modal and drop modal-local progress. `triggerManualPlanning()` sets `isLoading=true` in [usePlanningPrompt.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/hooks/usePlanningPrompt.js:131), and `AppShell` only renders the modal when `!planning.isLoading` in [AppShell.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/AppShell.jsx:98). That resets `assignedCount`, `step`, `dailyTasks`, row `isActioned`, row errors/loading, and the unused `skippedIds` state in [PlanningModal.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:28) and [PlanningTaskRow.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningTaskRow.jsx:50). Persisted task assignments are not lost, but the UI can show a false “you haven’t picked anything” warning after remount.

Low/Medium, pre-existing: `planning-complete` refreshes Today, Plan, and Calendar, but not Projects. Listeners exist in [TodayView.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/today/TodayView.jsx:172), [PlanBoard.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/plan/PlanBoard.jsx:271), and [CalendarView.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/calendar/CalendarView.jsx:63). `ProjectsView` loads task state in [ProjectsView.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/Projects/ProjectsView.jsx:59) but does not listen, so a planning session over `/projects` can leave project task buckets/attention filters stale until reload.

Advisory, pre-existing: concurrent row accepts can produce duplicate `sort_order`. `getMaxSortOrder()` reads current max then writes `max + 1` in [PlanningModal.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:69). The DB has indexes, not uniqueness, around sort order in [20260404000001_prioritisation_replacement.sql](/Users/peterpitcher/Cursor/OJ-Planner2.0/supabase/migrations/20260404000001_prioritisation_replacement.sql:217). Visual ordering remains stable enough because API ordering uses `sort_order, created_at` in [route.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:89), but semantic uniqueness is not guaranteed.

Low: `isManual` is now dead inside `PlanningModal`. It is still passed from [AppShell.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/AppShell.jsx:106) and destructured in [PlanningModal.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:24), but no longer affects behavior. Runtime-harmless, but structural noise and possible lint noise. Caveat: if manual weekly should differ from automatic Sunday-combined weekly, `isManual` may be the missing discriminator.

**Point-By-Point**

1. `assignedCount` is fine at modal level if it means “actions taken in this currently mounted wizard step.” It is not fine as durable window-level truth because the modal is intentionally or incidentally unmounted by the `isLoading` gate. If the modal remounts, assignments already persisted remain in the DB, but the counter resets and can trigger a false zero-assignment confirm.

2. The remount looks like a side effect of the loading gate, not an explicit state-machine transition. It does not lose persisted assignments because row actions call `updateTask()` immediately in [PlanningModal.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:83). It does lose local UI progress. Daily assigned tasks should mostly disappear after candidate refetch because daily candidates exclude `today`, but weekly due-this-week tasks can reappear because that candidate query excludes only `today/done`, not `this_week`, in [planning-candidates/route.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/planning-candidates/route.js:87).

3. The new `handleFinish` dependencies are semantically correct. It reads `assignedCount` and only `currentTasks.length`, so depending on `currentTasks.length` is enough. No other callback has stale-closure risk for those values: `handleAssign` uses functional state updates, and the other callbacks do not read `assignedCount` or `currentTasks`.

4. `isManual` is dead for current labeling. Leaving it is harmless at runtime, but I would either remove the prop plumbing or repurpose it deliberately if manual weekly should not run the Sunday-combined flow.

5. Counter behavior under update failure is correct. `assignedCount` increments only after `apiClient.updateTask()` succeeds. If the persisted update succeeds and a later local counter update is missed due to unmount, the only consequence is a possible false confirm, not data corruption.

6. Not resetting `sectionCounts` during Sunday weekly-to-daily transition is correct. Those counts represent absolute current `today` section load fetched from DB in [PlanningModal.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:45). Resetting to zero would make soft-cap warnings wrong. A re-fetch would be more accurate than a reset, but no reset is the right invariant.

7. The concurrent `sort_order` race is real and pre-existing. Claude’s change did not introduce it. It is an advisory unless the product requires unique ordering semantics.

8. Event coverage is partial. Today, Plan, and Calendar refresh on `planning-complete`; Projects does not. Also, because the event only fires from `onPlanningComplete()` in [usePlanningPrompt.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/hooks/usePlanningPrompt.js:163), closing the modal after persisted row actions but before Finish can leave currently mounted views stale. That is pre-existing.

9. `createPlanningSession` is idempotent. The API upserts on `user_id, window_type, window_date` in [planning-sessions/route.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/planning-sessions/route.js:72), backed by the unique constraint in [20260414000001_add_planning_tables.sql](/Users/peterpitcher/Cursor/OJ-Planner2.0/supabase/migrations/20260414000001_add_planning_tables.sql:2). The disabled button handles normal double-clicks; even if double-invoked, the backend result is benign.

10. The `\u2019` strings are build-safe in JSX. This file already uses Unicode/escaped Unicode for the week-range dash, saving ellipsis, and arrow in [PlanningModal.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:187).

11. Accessibility is acceptable. `window.confirm` is native modal UI. The hint `<p>` is visible descriptive text inside a Headless UI dialog; it does not need a role. If stronger screen-reader announcement is desired, it could be wired as dialog description, but this is not a structural blocker.

12. The factual-accuracy concern is mostly avoided by the branch. The “Finish Planning only records the session” copy is only in the daily hint in [PlanningModal.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:250). The weekly hint does not say that; it says to Accept/Defer/Skip and that nothing is added until an action is picked. For Sunday combined weekly, the button is “Next: Plan Monday →”, records the weekly session, and transitions. The weekly hint is acceptable, though “Nothing is added to this week until you Accept” would be more precise.
