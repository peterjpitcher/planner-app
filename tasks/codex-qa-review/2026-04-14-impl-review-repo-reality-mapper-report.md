**Findings**
1. The DB migration and the settings/API/UI do not agree on the `user_settings` schema. The migration creates `daily_planning_enabled`, `weekly_planning_enabled`, and `planning_snooze_until`, but the API and settings page read/write `daily_plan_start`, `daily_plan_end`, `weekly_plan_start`, and `weekly_plan_end` (`supabase/migrations/20260414000001_add_planning_tables.sql:17-24`, `src/app/api/user-settings/route.js:24-27`, `src/app/api/user-settings/route.js:103-113`, `src/app/settings/planning/PlanningSettingsClient.jsx:6-10`). On a fresh DB, `/api/user-settings` will query columns that do not exist.

2. Weekly window detection is wrong in two ways. `isInsideWindow()` treats `20:05 -> 20:00` as a simple wrap window, so on Sunday morning before the intended 20:05 start it still returns true; I spot-checked the current logic and `2026-04-19 10:00 BST` resolves to an active weekly window. Separately, `getActivePlanningWindow()` only considers weekly on Sunday or Monday-before-end, so the “revisit through the week” behavior never exists Tuesday-Saturday (`src/lib/planningWindow.js:30-42`, `src/lib/planningWindow.js:86-103`).

3. The Sunday combined flow is not implemented end-to-end. The hook fetches one candidate payload for the active window only (`src/hooks/usePlanningPrompt.js:48-55`). The modal then treats every weekly window as a two-step flow (`src/components/planning/PlanningModal.jsx:23-25`, `src/components/planning/PlanningModal.jsx:117-138`), but step 2 reuses the same `tasks` object and expects daily buckets that were never fetched (`src/components/planning/PlanningModal.jsx:141-143`, `src/components/planning/PlanningModal.jsx:175-184`). In practice, the Monday daily step has no `dueTomorrow`/`undatedThisWeek` data.

4. The cron reschedule is internally inconsistent and will run early in GMT. `vercel.json` schedules both `18:55` and `19:55` UTC for both demotion routes (`vercel.json:17-30`). The route guards now allow London hours `18-20` (`src/app/api/cron/demote-today-tasks/route.js:27-33`, `src/app/api/cron/demote-week-tasks/route.js:34-40`), and `claimCronRun()` is once per `operation + run_date` (`src/lib/cronAuth.js:80-105`). In winter, the `18:55 UTC` invocation is `18:55 London`, so it claims the run an hour before the intended `19:55 London` slot.

5. “New tasks” and banner messaging are not tracking real “new” work. After any recorded session, the hook sets `hasNewTasks` true if *any* candidates still exist (`src/hooks/usePlanningPrompt.js:57-63`), so skipped overdue items will be labeled as “new tasks.” The banner also uses `totalCandidates` across all buckets (`src/hooks/usePlanningPrompt.js:114-116`, `src/components/planning/PlanningBanner.jsx:20-26`), so “You have N tasks due tomorrow” can include overdue and undated-this-week tasks.

6. The planned/revisit banner disappears when the user fully clears the queue. AppShell only renders the banner when `planning.totalCandidates > 0` (`src/components/layout/AppShell.jsx:81-90`), which suppresses the spec’s “Tomorrow’s planned / Week planned — Revisit” state once the candidate query is empty.

7. The planning UI still hits `/api/tasks` GET and therefore still triggers Office365 auto-sync. The modal uses `apiClient.getTasks()` for section counts and max sort order (`src/components/planning/PlanningModal.jsx:34-70`), and `/api/tasks` fires `maybeAutoSyncOffice365()` on every GET (`src/app/api/tasks/route.js:35-49`). So the feature is not side-effect free even though `/api/planning-candidates` itself is.

8. Task rows optimistically mark themselves actioned without awaiting mutation success. `PlanningTaskRow` calls `onAssign` / `onDefer` and immediately flips local UI state (`src/components/planning/PlanningTaskRow.jsx:64-87`), while the modal just logs failures (`src/components/planning/PlanningModal.jsx:72-93`, `src/components/planning/PlanningModal.jsx:99-115`). A failed PATCH can still look “done,” and `Finish Planning` can be submitted before all task mutations settle (`src/components/planning/PlanningModal.jsx:117-138`).

9. The defer backlog rule is wrong for daily planning. It compares the new date to `windowDate + 6` (`src/components/planning/PlanningModal.jsx:101-109`). In daily mode `windowDate` is tomorrow, not Monday-of-current-week, so deferring from a Thursday prompt to next Monday can incorrectly stay out of `backlog`.

10. Settings invalidation is half-wired. The hook exposes `refreshSettings()` but nothing calls it (`src/hooks/usePlanningPrompt.js:100-102`, `src/hooks/usePlanningPrompt.js:131`), and the settings page only saves via `apiClient.updateUserSettings()` (`src/app/settings/planning/PlanningSettingsClient.jsx:45-56`). Updated window times will not be seen by the mounted hook until a tab refocus triggers cache reset.

**Repo Reality Map**
- Actual control flow is: `AppShell` mounts `usePlanningPrompt` (`src/components/layout/AppShell.jsx:26`, `src/components/layout/AppShell.jsx:81-105`) -> hook fetches `/api/user-settings`, computes window via `getActivePlanningWindow`, then calls `/api/planning-sessions` and `/api/planning-candidates` (`src/hooks/usePlanningPrompt.js:28-76`) -> AppShell renders `PlanningBanner` and `PlanningModal`.
- Task mutation flow is: `PlanningTaskRow` -> `PlanningModal.handleAssign/handleDefer` -> `apiClient.updateTask()` (`src/components/planning/PlanningTaskRow.jsx:64-87`, `src/components/planning/PlanningModal.jsx:72-115`, `src/lib/apiClient.js:169-185`) -> existing `PATCH /api/tasks`.
- Session completion flow is: modal `handleFinish()` -> `apiClient.createPlanningSession()` (`src/components/planning/PlanningModal.jsx:117-138`, `src/lib/apiClient.js:322-328`) -> `POST /api/planning-sessions`, then `onPlanningComplete()` dispatches `planning-complete` (`src/hooks/usePlanningPrompt.js:104-112`). `TodayView`, `PlanBoard`, and `CalendarView` all listen and refetch (`src/components/today/TodayView.jsx:172-177`, `src/components/plan/PlanBoard.jsx:271-276`, `src/components/calendar/CalendarView.jsx:63-68`).
- Auth pattern is consistent: all three new API routes call `getAuthContext(request)` and scope service-role queries with `session.user.id` (`src/app/api/planning-candidates/route.js:8-29`, `src/app/api/planning-sessions/route.js:6-31`, `src/app/api/user-settings/route.js:16-28`). Validation is manual string/regex checking, not Zod.
- The settings UX is a separate route, not an addition to an existing consolidated settings page: `/settings/planning` is its own page and is linked directly from the sidebar (`src/app/settings/planning/page.js:1-9`, `src/components/layout/Sidebar.jsx:20-29`).
- There is also schema drift in `planning_sessions`: the migration adds `started_at`, `dismissed_at`, `tasks_promoted`, and `tasks_added`, but the code only ever reads/writes `window_type`, `window_date`, and `completed_at` (`supabase/migrations/20260414000001_add_planning_tables.sql:2-13`, `src/app/api/planning-sessions/route.js:25-31`, `src/app/api/planning-sessions/route.js:64-76`).

No tests were run; this was a static implementation review plus a direct spot-evaluation of the current planning-window logic.
