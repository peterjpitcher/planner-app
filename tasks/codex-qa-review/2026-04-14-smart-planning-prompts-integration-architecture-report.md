**Findings**
- `ARCH-001 (High)` The spec’s candidate-task fetch does not fit the current `/api/tasks` contract. The spec says `usePlanningPrompt` can fetch planning candidates through the existing tasks API [design spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:121), but the real route only filters by `projectId`, `state`/`states`, `completedSince`, `limit`, and `offset` [tasks route](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:53), [tasks route](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:74). It cannot express “due tomorrow”, “overdue but not today/done”, or “undated this_week”, and it defaults to 100 rows [tasks route](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:58). As written, the hook would have to over-fetch and filter client-side, which is both incomplete and expensive. This needs either a dedicated planning-candidates query or a server-side extension of `/api/tasks`.

- `ARCH-002 (High)` Planning state cannot be made the hook’s job alone without breaking the current ownership model. `TodayView`, `PlanBoard`, and `CalendarView` each own their own task collections, optimistic updates, and drawer state locally [TodayView.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/today/TodayView.jsx:85), [PlanBoard.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/plan/PlanBoard.jsx:179), [CalendarView.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/calendar/CalendarView.jsx:25). A modal mounted above them in `AppShell` will mutate tasks out-of-band; there is no shared task store, query cache, or event bus to refresh the active view. The clean split is: AppShell owns prompt visibility/session metadata, the modal owns its temporary candidate list, and the views remain the source of truth for rendered task collections. That requires a narrow invalidation/provider mechanism, not a global “planning hook owns tasks” design.

- `ARCH-003 (High)` Reusing `apiClient.updateTask()` creates real coupling to the existing task mutation path and its side effects. The client always goes through `PATCH /api/tasks` [apiClient.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/lib/apiClient.js:169), [tasks route](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:177), and `taskService.updateTask()` resets `entered_state_at` on state change, manages `completed_at`, touches related projects, and triggers Office365 sync per task [taskService.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/services/taskService.js:181), [taskService.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/services/taskService.js:202), [taskService.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/services/taskService.js:237), [taskService.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/services/taskService.js:244). It does not assign a new `sort_order` on moves into `today`/`this_week`, while the UI sorts by `sort_order` [TodayView.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/today/TodayView.jsx:59). So the planning modal inherits undefined placement semantics and N server-side side effects for N task clicks. If this feature stays mutation-heavy, a batch planning endpoint is architecturally cleaner.

- `ARCH-004 (Medium)` The right mount point is `AppShell`, not the root layout or the individual views. `src/app/layout.js` is just the wrapper; `AppShell` is the client boundary inside providers [layout.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/layout.js:16), [AppShell.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/AppShell.jsx:12). The banner should mount inside `<main>` below `TabBar` and above `{children}` [AppShell.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/AppShell.jsx:71); the modal should mount as a fixed sibling in `AppShell` with a z-index above the mobile nav overlay/header/QuickCapture [AppShell.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/AppShell.jsx:59), [Header.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/Header.jsx:8), [QuickCapture.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/shared/QuickCapture.jsx:122). That is not a boundary violation. It becomes one only if the spec expects each view to render its own banner. One concrete trap: `TAB_ROUTES` currently omits `/calendar` even though the tab bar includes it [AppShell.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/AppShell.jsx:10), [TabBar.jsx](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/TabBar.jsx:8), so any reuse of current “main view” gating will miss Calendar.

- `ARCH-005 (Medium)` The new API routes should follow the existing auth/service-role/manual-scoping pattern. The dominant route shape is `try/catch`, `getAuthContext(request)`, `getSupabaseServiceRole()`, hard-pin queries to `session.user.id`, and return `NextResponse.json(...)` [tasks route](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:12), [tasks route](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:28), [tasks route](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:51). `/api/planning-sessions` and `/api/user-settings` should use the same pattern, ideally with small service modules rather than inline Supabase logic. `PUT /api/user-settings` is not wrong, but the repo’s update routes are mostly `PATCH`; if you want to match existing conventions, `PATCH` is the closer fit.

- `ARCH-006 (Medium)` The proposed tables are directionally consistent with the repo’s schema style, but the spec is missing several invariants that matter here. Recent tables use UUID PKs, `auth.users` FKs, `timestamptz`, user-scoped RLS, and `updated_at` triggers/check constraints where appropriate [initial schema](/Users/peterpitcher/Cursor/OJ-Planner2.0/supabase/migrations/20250707_initial_schema.sql:92), [prioritisation migration](/Users/peterpitcher/Cursor/OJ-Planner2.0/supabase/migrations/20260404000001_prioritisation_replacement.sql:15), [RLS migration](/Users/peterpitcher/Cursor/OJ-Planner2.0/supabase/migrations/20250904_performance_and_rls.sql:36). `planning_sessions` needs at least a unique key on `(user_id, window_type, window_date)` plus an index for lookups; `user_settings` needs an `updated_at` trigger and RLS; `window_type` should get a `CHECK` constraint like other text enums [prioritisation migration](/Users/peterpitcher/Cursor/OJ-Planner2.0/supabase/migrations/20260404000001_prioritisation_replacement.sql:151). Also, the spec says `planning_sessions.created_at` means “when the session was started” [design spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:34), but the only write happens on Finish Planning [design spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:89), so that column definition and API flow do not currently match.

- `ARCH-007 (Medium)` The UI spec still references a removed field. `PlanningModal` says each row shows a “priority indicator” [design spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:85), but `priority` was removed from `tasks` in the state-model migration [prioritisation migration](/Users/peterpitcher/Cursor/OJ-Planner2.0/supabase/migrations/20260404000001_prioritisation_replacement.sql:137), is absent from task selects [taskService.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/services/taskService.js:38), and is stripped from updates client-side [apiClient.js](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/lib/apiClient.js:170). That part of the spec does not fit the current architecture.

Net: the feature can fit cleanly, but only if `AppShell` owns the cross-cutting UI, a dedicated planning-query path is added, and the spec defines how active views are invalidated after modal-driven task mutations. As written, the biggest gaps are candidate fetching and cross-view state sync.
