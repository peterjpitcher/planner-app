**Assumption Breaker Report**

**AB-001 High**  
The spec assumes the hook can fetch planning candidates from the existing `/api/tasks` endpoint “with appropriate filters” ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:121)). That endpoint does not implement due-date or date-range filtering; it only parses `projectId`, `state`, `states`, `completedSince`, `limit`, and `offset` ([GET /api/tasks](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:53)). The client already appends `range`, `days`, and `includeOverdue`, but the server ignores them ([apiClient](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/lib/apiClient.js:87)). So the spec is not “reusing existing filters”; it is either missing required server work or silently forcing a full client-side scan/pagination of non-done tasks, which is capped at `100` by default and `200` max per request and also triggers Office365 auto-sync side effects on every tasks GET ([GET /api/tasks](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:35)).

**AB-002 High**  
Per-user configurable planning times conflict directly with the existing demotion model. The spec adds `user_settings` time fields and says “no new cron jobs” ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:36), [spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:134)), but the actual demotion jobs are hard-coded to London hour `20` ([demote today](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/cron/demote-today-tasks/route.js:26), [demote week](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/cron/demote-week-tasks/route.js:33)) and scheduled only at `19:00`/`20:00` UTC to catch DST ([vercel.json](/Users/peterpitcher/Cursor/OJ-Planner2.0/vercel.json:17)). They also resolve a single env-configured user, not arbitrary per-user schedules ([dailyTaskEmailService](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/services/dailyTaskEmailService.js:72)). If a user sets `18:00` or `22:00`, the planning window and demotion semantics immediately diverge.

**AB-003 High**  
“Check current London time on every page load/navigation” is underspecified and not robust in this app shape ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:21), [spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:113)). `src/app/layout.js` is a server component, so the hook has to live in the persistent client `AppShell`, which means it will not naturally rerun on client navigation unless it keys off pathname changes ([layout](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/layout.js:16), [AppShell](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/AppShell.jsx:12)). Also, `getStartOfTodayLondon()` only gives London midnight, not arbitrary 20:05-20:00 window membership ([dateUtils](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/lib/dateUtils.js:138)), while existing date helpers and quick picks still use local browser time ([dateUtils](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/lib/dateUtils.js:20), [dateUtils](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/lib/dateUtils.js:75)). The real issue is not browser timezone detection; it is trusting client clock/time at all for a London-authoritative gating rule.

**AB-004 High**  
The spec says the feature is “entirely client-driven” and existing demote crons can “continue independently” ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:134)). They are not independent. At London hour 20 the server mutates `today -> this_week` and `this_week -> backlog` in place ([demote today](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/cron/demote-today-tasks/route.js:63), [demote week](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/cron/demote-week-tasks/route.js:70)), while the planning window opens at 20:05 ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:16)). A late cron run or slow loop means the prompt can read a different candidate set at 20:05 than at 20:06. Sunday is the worst case because the weekly prompt starts right after the weekly demotion boundary.

**AB-005 Medium**  
The “new tasks added mid-window” detection logic is not reliable as written. The spec says to compare surfaced count against tasks already in TODAY ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:145)), but `planning_sessions` stores no surfaced task IDs or per-task outcomes, only window metadata ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:25)). Count-based detection breaks on net-zero changes, skipped tasks, deferred tasks, completed tasks, and it does not even generalize to weekly planning because weekly “Accept” moves tasks to `this_week`, not `today` ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:72)). Existing task metadata like `entered_state_at` only tracks state transitions, not session baselines ([taskService](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/services/taskService.js:181)).

**AB-006 Medium**  
The UI ownership model does not fit the current shell. The spec says the hook is used “in the app layout” and both modal and banner consume it ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:113)), but `AppShell` only renders shared chrome, route content, and `QuickCapture`; there is no existing global modal/banner host ([AppShell](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/AppShell.jsx:53)). The only current planning-prompt-like pattern is a localStorage-gated banner owned by `TodayView` ([TodayView](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/today/TodayView.jsx:141), [TodayView](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/today/TodayView.jsx:468)), and modal/drawer ownership is page-local in views like `TodayView` and `PlanBoard` ([TodayView](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/today/TodayView.jsx:588), [PlanBoard](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/plan/PlanBoard.jsx:685)). There is also a route-fit gap: AppShell’s shared “tab route” logic includes `/today`, `/plan`, `/projects`, and `/ideas`, but not `/calendar`, even though the spec wants the banner on calendar too ([AppShell](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/layout/AppShell.jsx:10)).

**AB-007 Medium**  
The partial-completion claim is only half-true. The exact example of “assigned undated THIS_WEEK tasks reappearing in the Undated THIS_WEEK bucket” does not hold, because that bucket requires `state = this_week`, so tasks moved to `today` will drop out ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:57)). But the spec still overstates “only the remaining unactioned tasks” on reopen ([spec](/Users/peterpitcher/Cursor/OJ-Planner2.0/docs/superpowers/specs/2026-04-14-smart-planning-prompts-design.md:144)): `Skip` leaves a task unchanged, and `Defer` may still leave it inside the active window. Because `planning_sessions` does not persist per-task outcomes, reopen cannot distinguish “already reviewed and intentionally skipped/deferred” from “never handled”.

Net: the spec is underestimating required server work. It needs a server-authoritative planning query, a server-authoritative window decision, and a clearer contract for UI ownership and per-session task baselines before it fits this codebase cleanly.
