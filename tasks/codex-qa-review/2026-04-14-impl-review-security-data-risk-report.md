**Findings**

- `SEC-001 (High)` `user_settings` is schema-incompatible with the code that reads and writes it. The migration creates `daily_planning_enabled`, `weekly_planning_enabled`, and `planning_snooze_until` in [20260414000001_add_planning_tables.sql](/Users/peterpitcher/Cursor/OJ-Planner2.0/supabase/migrations/20260414000001_add_planning_tables.sql:17), but the API selects and upserts `daily_plan_start`, `daily_plan_end`, `weekly_plan_start`, and `weekly_plan_end` in [user-settings GET](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/user-settings/route.js:24) and [user-settings PATCH](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/user-settings/route.js:103). `usePlanningPrompt()` blocks on `getUserSettings()` before it can compute any active window in [usePlanningPrompt](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/hooks/usePlanningPrompt.js:30), so a DB built from the checked-in migration turns settings reads/writes into 500s and silently disables prompt detection.

- `SEC-002 (High)` The UI can record a planning session even when task mutations failed. `PlanningTaskRow` marks rows as actioned immediately after calling the async handlers in [PlanningTaskRow](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningTaskRow.jsx:64), while `PlanningModal` only logs assignment/defer failures in [handleAssign](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:72) and [handleDefer](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:99). `handleFinish()` then still posts `/api/planning-sessions` in [PlanningModal](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:117) and the server upserts completion in [planning-sessions POST](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/planning-sessions/route.js:63). A transient PATCH failure can therefore leave tasks unmoved while the server suppresses the prompt for that window.

- `SEC-003 (Medium)` Opening the planning modal has hidden external-sync side effects. The modal calls `apiClient.getTasks()` for section counts and sort-order lookups in [PlanningModal](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:34) and [PlanningModal](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:57). Every `/api/tasks` GET kicks off `maybeAutoSyncOffice365()` in [tasks GET](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/tasks/route.js:35). So a “read-only” planning prompt can unexpectedly spend Office365 credentials and mutate synced task data just by being opened.

- `SEC-004 (Medium)` `windowDate` validation is only shape-based, not semantic. Both planning routes accept anything matching `YYYY-MM-DD` in [planning-candidates validation](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/planning-candidates/route.js:20) and [planning-sessions validation](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/planning-sessions/route.js:56). Impossible dates like `2026-02-31` pass. The weekly candidates route then feeds that value into JS date math in [planning-candidates weekly window](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/planning-candidates/route.js:79), and the sessions route sends it to a `date` column in [planning-sessions upsert](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/planning-sessions/route.js:66). Result: bad input can become the wrong week or a 500 instead of a clean 400.

- `SEC-005 (Medium)` The Sunday combined flow can write a daily completion record without ever loading daily candidates. The hook fetches one candidate payload for the active window in [usePlanningPrompt](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/hooks/usePlanningPrompt.js:49). The modal treats weekly windows as the combined flow in [PlanningModal](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:23), switches from weekly to daily in [PlanningModal](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:120), but the daily step reads `dueTomorrow` / `undatedThisWeek` from the same `tasks` object in [PlanningModal](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:141). If only weekly candidates were loaded, the user can still finish and create the daily session in [PlanningModal](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/components/planning/PlanningModal.jsx:129). That creates false “planned” state for unseen Monday tasks.

- `SEC-006 (Low)` `PATCH /api/user-settings` validates partial updates against defaults, not the stored row, so API callers can still persist zero-length windows. The comparison uses default values in [user-settings validation](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/user-settings/route.js:72), then performs a partial upsert in [user-settings upsert](/Users/peterpitcher/Cursor/OJ-Planner2.0/src/app/api/user-settings/route.js:103). If a row already has `daily_plan_end = 19:00`, a PATCH with only `daily_plan_start = 19:00` passes validation and saves an unusable window.
